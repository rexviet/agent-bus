1. **Create `runSlot` async function** inside `runWorkerCommand`, after daemon startup and before the main execution block:

```typescript
async function runSlot(slotIndex: number): Promise<void> {
  const slotWorkerId = `${workerId}/${slotIndex}`;
  let slotInFlight = false;

  while (!stopController.requested) {
    // Mutex: serialize claim path with in-flight backpressure check
    const result = await claimMutex.run(async () => {
      if (inFlightCount >= concurrency) return null;
      inFlightCount++;
      slotInFlight = true;
      try {
        return await daemon.runWorkerIteration(slotWorkerId, leaseDurationMs, retryDelayMs);
      } catch (error) {
        // Decrement on claim/iteration failure
        inFlightCount--;
        slotInFlight = false;
        throw error;
      }
    });

    // If result is null because of backpressure (inFlightCount >= concurrency), sleep
    if (result === undefined) {
      // mutex.run returned null from backpressure check
      await Promise.race([sleep(pollIntervalMs), stopController.waitForStop()]);
      continue;
    }

    if (result === null) {
      // claim returned null — nothing to claim
      inFlightCount--;
      slotInFlight = false;
      idlePolls += 1;

      if (once) {
        stopController.request("once");
        if (slotIndex === 0) {
          writeWorkerIdleText(io.stdout, workerId);
        }
        break;
      }

      await Promise.race([sleep(pollIntervalMs), stopController.waitForStop()]);
      continue;
    }

    // Successfully processed a delivery
    inFlightCount--;
    slotInFlight = false;
    idlePolls = 0;
    processedDeliveries += 1;
    writeWorkerExecutionText(io.stdout, `${workerId}/${slotIndex}`, result);

    if (once) {
      stopController.request("once");
      break;
    }
  }
}
```

**IMPORTANT design note:** The above is conceptual. The key constraint is:
- The `inFlightCount` check AND the `claim()` call must BOTH be inside `claimMutex.run()` to prevent TOCTOU (per CONTEXT.md locked decision and RESEARCH.md Pitfall 3).
- `inFlightCount++` happens inside the mutex BEFORE `runWorkerIteration` returns (since `runWorkerIteration` includes both claim and execution). Actually, re-examine: `runWorkerIteration` does claim + full execution. If the entire `runWorkerIteration` is inside the mutex, that serializes execution (anti-pattern from RESEARCH.md).

**Correct approach (from RESEARCH.md Pattern 1):** Split the claim from execution. However, `daemon.runWorkerIteration` atomically claims + executes. We need to restructure slightly:

- Option A: Accept that the claim part of `runWorkerIteration` is fast (single SQLite write) and the rest is agent execution. Wrapping only the claim inside mutex requires splitting `runWorkerIteration`. But the daemon API bundles them.
- Option B (recommended): Use a different tracking approach. The mutex wraps only: `if (inFlightCount >= concurrency) return null; inFlightCount++; return "proceed";`. Then OUTSIDE the mutex, call `runWorkerIteration`. The increment inside mutex prevents TOCTOU. The `finally` block decrements after iteration completes.

**Implement Option B:**

```typescript
const claimMutex = createMutex();
let inFlightCount = 0;

async function runSlot(slotIndex: number): Promise<void> {
  const slotWorkerId = `${workerId}/${slotIndex}`;

  while (!stopController.requested) {
    // Acquire slot capacity inside mutex (prevents TOCTOU)
    const canProceed = await claimMutex.run(async () => {
      if (inFlightCount >= concurrency) return false;
      inFlightCount++;
      return true;
    });

    if (!canProceed) {
      await Promise.race([sleep(pollIntervalMs), stopController.waitForStop()]);
      continue;
    }

    // Slot capacity acquired — run iteration outside mutex
    try {
      const result = await daemon.runWorkerIteration(slotWorkerId, leaseDurationMs, retryDelayMs);

      if (result) {
        processedDeliveries += 1;
        writeWorkerExecutionText(io.stdout, slotWorkerId, result);

        if (once) {
          stopController.request("once");
          break;
        }
        continue; // Skip sleep, immediately try next claim
      }

      // Nothing to claim
      idlePolls += 1;
      if (once) {
        stopController.request("once");
        if (slotIndex === 0) writeWorkerIdleText(io.stdout, workerId);
        break;
      }

      await Promise.race([sleep(pollIntervalMs), stopController.waitForStop()]);
    } finally {
      inFlightCount--;
    }
  }
}
```

When stop is requested (from signal handler), start a drain timeout that races against slot completion:

```typescript
// Before launching slots, set up a drain timeout listener
stopController.waitForStop().then(async () => {
  // Stop was requested — start drain timeout
  const drainTimer = sleep(drainTimeoutMs);
  await drainTimer;
  // If we get here, drain timed out — force exit
  // The slotPromises may still be running, but we let process.exit handle it

    logger.warn({ event: "drain.timeout", inFlightCount, drainTimeoutMs });
  }
});
```

Actually, the cleaner approach: race the slot promises against the drain timeout AFTER stop is requested:

```typescript
const slotPromises = Array.from({ length: concurrency }, (_, i) => runSlot(i));

// Wait for either: all slots complete naturally, or stop+drain timeout
const allDone = Promise.allSettled(slotPromises);

await allDone; // Slots exit their while-loop when stopController.requested becomes true

// But if drain takes too long, we need a timeout. Handle this by:
// When stop is requested, start a drain timer. The slots will notice stopController.requested
// and stop claiming. In-flight iterations will complete. If they don't complete within
// drainTimeoutMs, we proceed to cleanup anyway.

// Better: wrap the await in a race
```

**Final recommended structure:**

```typescript
try {
  const slotPromises = Array.from({ length: concurrency }, (_, i) => runSlot(i));

  // Wait for all slots, but with drain timeout after stop is requested
  await new Promise<void>((resolve) => {
    // Track when all slots finish
    Promise.allSettled(slotPromises).then(() => resolve());

    // Set up drain timeout: only activates after stop is requested
    stopController.waitForStop().then(() => {
      drainedDeliveries = inFlightCount;
      sleep(drainTimeoutMs).then(() => {
        // Drain timed out — resolve anyway, let cleanup proceed
        if (inFlightCount > 0) {
          logger.warn({ event: "drain.timeout", inFlightCount, drainTimeoutMs });
        }
        resolve();
      });
    });
  });

  return 0;
} finally {
  stopController.request(stopController.reason);
  unregisterSignals();
  await daemon.stop();
  writeWorkerStoppedText(io.stdout, {
    workerId,
    processedDeliveries,
    drainedDeliveries,
    idlePolls,
    reason: stopController.reason
  });
}
```

**Key behaviors to preserve:**
- `--once` mode: first slot that claims a delivery processes it, then requests stop. Other slots see stop and exit. With no delivery available, slot 0 writes idle text.
- Error in `runWorkerIteration`: write error to stderr, return exit code 1 (same as current behavior). Consider: with concurrent slots, one slot error should not crash others. Wrap per-slot errors: if one slot throws, let others continue. Log the error but don't return 1 immediately. Only return 1 if ALL slots encounter errors.
- `drainedDeliveries` is the in-flight count at the moment stop is requested (snapshot of deliveries that needed draining).
