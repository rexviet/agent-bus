<!-- SSE event contract from plan 01/02 -->
SSE endpoint sends these event types:
```
event: snapshot
data: { "runs": OperatorRunSummary[], "approvals": PendingApprovalView[], "failures": FailureDeliveryView[] }

event: delivery.state_changed
data: { "deliveryId": string, "runId": string, "agentId": string, "oldState": string|undefined, "newState": string }

event: approval.created
data: { "approvalId": string, "eventId": string, "runId": string, "topic": string }

event: approval.decided
data: { "approvalId": string, "status": "approved"|"rejected", "decidedBy": string }

event: event.published
data: { "eventId": string, "runId": string, "topic": string }
```

API endpoints:
```
GET /api/runs -> OperatorRunSummary[]
GET /api/runs/:runId -> OperatorRunDetail (with events, approvals, deliveries arrays)
GET /api/approvals -> PendingApprovalView[]
GET /api/failures -> FailureDeliveryView[]
```

OperatorRunSummary shape:
```typescript
{ runId, status, metadata, createdAt, updatedAt, eventCount, approvalCount, deliveryCount, deliveryStatusCounts: { pendingApproval, ready, leased, retryScheduled, completed, deadLetter, cancelled, total }, latestEventAt? }
```

PendingApprovalView shape:
```typescript
{ approvalId, eventId, runId, topic, status, requestedAt, producerAgentId, approvalStatus, deliveryCount }
```

FailureDeliveryView shape:
```typescript
{ deliveryId, eventId, runId, topic, agentId, status, errorMessage, attemptCount, maxAttempts, producerAgentId, producerRuntime, ... }
```

  <action>
    1. Create `src/daemon/dashboard-html.ts` exporting `getDashboardHtml(): string` that returns a template literal containing the entire dashboard as a single HTML page with inline `<style>` and `<script>`.

    2. **HTML Structure** (per locked decisions in CONTEXT.md):
       - Minimal header: "Agent Bus" title on left, SSE connection indicator on right
       - "Show all sections" toggle checkbox (default unchecked = empty sections hidden)
       - Three sections in attention-first order:
         a. **Pending Approvals** — list of pending approval rows
         b. **Failures** — list of dead-letter/retry deliveries
         c. **Runs** — list of run rows, most recent at top
       - Each section has a header with count badge
       - Empty sections hidden by default (CSS class toggled by JS based on checkbox)

    3. **Run rows** (per locked decisions):
       - One line per run: truncated run ID (first 8 chars), status badge, delivery count (completed/total), relative timestamp
       - Clickable — expands inline (no page navigation)
       - Expanded view shows all subsections visible (not individually collapsible):
         - Deliveries: table with agentId, state dot + label, timing
         - Events: list with eventId, topic, occurredAt
         - Approvals: list with approvalId, status, requestedAt
         - Error messages for failed deliveries
       - Collapse by clicking the run row header again

    4. **Failure rows**:
       - Show deliveryId (truncated), agentId, error summary (first 80 chars), attempt count
       - Clicking scrolls to parent run in Runs section and expands it with the failed delivery highlighted (add a CSS class for highlight)

    5. **CSS** (per locked decisions — dark terminal aesthetic):
       - Background: #1a1a2e body, #1e293b for cards/rows
       - Border: subtle dark borders (e.g., #2d3748)
       - Font: monospace, color #e0e0e0
       - Status dots (small circles via CSS):
         - completed: #4ade80 (green)
         - leased/in_progress: #60a5fa (blue)
         - retry_scheduled/attention_required: #fbbf24 (amber)
         - dead_letter: #f87171 (red)
         - ready: #9ca3af (gray)
         - pending_approval: gray outline (border only)
         - cancelled: dim (#6b7280)
       - Status badges for runs use same color coding
       - Header connection indicator: green dot for live, gray for disconnected, pulsing for reconnecting
       - Flash animation: amber glow keyframe, 1-2s duration, applied via class then removed
       - Responsive: single column, max-width ~1200px centered

    6. **Inline JS** (`<script>` block):
       a. **SSE connection** with manual reconnect and exponential backoff:
          ```javascript
          function connect() {
            const es = new EventSource("/events");
            let retryMs = 3000;

            es.addEventListener("snapshot", (e) => {
              const data = JSON.parse(e.data);
              renderAll(data);
              retryMs = 3000;
              setIndicator("live");
            });

            es.addEventListener("delivery.state_changed", (e) => {
              const p = JSON.parse(e.data);
              updateDeliveryInRun(p);
              flashRow(p.deliveryId);
            });

            es.addEventListener("approval.created", (e) => {
              const p = JSON.parse(e.data);
              refreshApprovals();
            });

            es.addEventListener("approval.decided", (e) => {
              const p = JSON.parse(e.data);
              refreshApprovals();
              refreshRuns(); // approval status change affects run status
            });

            es.addEventListener("event.published", (e) => {
              const p = JSON.parse(e.data);
              refreshRuns(); // new event may mean new run
            });

            es.onerror = () => {
              setIndicator("disconnected");
              es.close();
              setTimeout(() => {
                setIndicator("reconnecting");
                connect();
                retryMs = Math.min(retryMs * 2, 30000);
              }, retryMs);
            };
          }
          ```

       b. **Refresh functions**: `refreshRuns()`, `refreshApprovals()`, `refreshFailures()` that `fetch` the API endpoints and re-render the respective sections. Called on SSE reconnect (after snapshot) and for events that need full re-query.

       c. **renderAll(snapshot)**: Full page render from snapshot data. Renders all three sections.

       d. **Relative timestamps**: `function relativeTime(isoString)` returning "just now", "Xm ago", "Xh ago", "Xd ago". Updated every 30s via `setInterval`.

       e. **Inline expand**: Click handler on run rows toggles a details div. When expanding, fetch `/api/runs/{runId}` for full detail (deliveries, events, approvals) and render inline.

       f. **Failure cross-reference**: Click handler on failure rows finds the parent run element by runId, scrolls into view, expands it, and adds a highlight class to the failed delivery row.

       g. **Section visibility**: Toggle handler for "Show all sections" checkbox. When unchecked (default), sections with zero items get `display: none`.

       h. **Connection indicator**: `setIndicator(state)` function updates the header indicator dot and label.

       i. **Flash animation**: `flashRow(elementId)` adds a CSS class, removes it after 1.5s via setTimeout.

    7. In `src/daemon/dashboard-server.ts`:
       - Replace the placeholder `GET /` handler with:
         ```typescript
         import { getDashboardHtml } from "./dashboard-html.js";
         app.get("/", (c) => {
           return c.html(getDashboardHtml());
         });
         ```

    8. Add test to `test/daemon/dashboard-server.test.ts`:

       - Test: Response body contains `EventSource` (SSE client present)
       - Test: Response body contains the status color values (#4ade80, #f87171, etc.)

    9. IMPORTANT: The HTML template is a single TypeScript template literal. No separate files. No `fs.readFileSync`. This ensures DASH-08 compliance (no build pipeline, no static file serving).
