    4. In `startDaemon()`, after `migrateDatabase(database)` and before creating stores: start the MCP server:
       ```
       const mcpServer = await createMcpServer({
         publishEvent: (envelope) => publishEvent({
           database, manifest, runStore, eventStore, deliveryStore, dispatcher, envelope
         }),
         ...(options.mcpPort !== undefined ? { port: options.mcpPort } : {})
       });
       ```
       Note: The stores (runStore, eventStore, etc.) must be created BEFORE the MCP server because `publishEvent` closes over them. Reorder if needed: create stores first, then start MCP server.
       Actually, looking at the code, stores are created after database. The publishEvent callback will be called later (during requests), not during construction. So the stores just need to exist by the time a tool call comes in. Since the MCP server starts before the poll loop, and stores are created before the MCP server call, this is fine. Reorder to: open DB -> migrate -> create stores -> create dispatcher -> start MCP server -> create adapter worker.
    5. Thread `mcpUrl` into `AdapterWorkerOptions`: add `mcpUrl: mcpServer.url` to the `adapterWorkerOptions` object.
    6. In `stop()`: add `await mcpServer.stop();` AFTER `recoveryScan.stop()` and BEFORE `database.close()`. This matches the shutdown order: stop claiming -> drain in-flight -> close MCP server -> close SQLite.
    7. Return `mcpUrl: mcpServer.url` on the daemon object.

    **src/daemon/adapter-worker.ts:**
    1. Add `readonly mcpUrl?: string;` to `AdapterWorkerOptions` interface.
    2. In `buildAdapterCommand()` call inside `runIteration()`: add `mcpUrl: options.mcpUrl` to the input. The `BuildAdapterCommandInput` interface needs extending too (see registry.ts changes).

    3. The `vendorInput` object in `buildAdapterCommand()` already spreads `baseEnvironment: buildBaseEnvironment(input)`, so vendor commands automatically get the new env var.

    IMPORTANT: The `publishEvent` callback passed to `createMcpServer` must reference the stores that are already initialized. Review the variable ordering in `startDaemon()` carefully. The callback is a closure -- it captures `database`, `manifest`, `runStore`, `eventStore`, `deliveryStore`, `dispatcher`. All must be defined before `createMcpServer()` is called. Currently the code creates stores before the adapter worker, so insert MCP server creation AFTER dispatcher creation and BEFORE adapterWorkerOptions construction.
